# Web2Print — Work Log (2026-06-18)

Summary of the day's work. All changes are on branch `security/4over-env-vars`,
pushed to the testing remote `web2print-clone` (never to `origin`).

Scope: integrate the **4over** print provider end-to-end and make the print
storefront mirror the reference (OnPrintShop / fourprintshop). Sandbox 4over
API is used so test orders are possible.

---

## 1. Security — API credentials → environment variables
- Removed hardcoded 4over fallback credentials (`web2printusa` / `3KHNXZFZ`)
  from `lib/4over/client.ts`; credentials now resolve lazily from env at
  request time via `getCredentials()` (clear error if missing — no silent
  bad-key usage, no build crash).
- Added `.env.example` template (all required vars, server-only vs
  `NEXT_PUBLIC_` clearly separated) and a `.gitignore` exception so the
  template is tracked but real `.env.local` is not.
- ⚠️ The old secret is still in git history — rotate in the 4over dashboard
  if it is a live credential.

## 2. 4over pricing — live calculator
- **Turnaround bug fixed** (`/api/4over/baseprices`): 4over's `baseprices`
  rows have no turnaround, and the nested `productoptiongroups` /
  `turnaroundtimes` endpoints 404. The turnaround options actually live in
  the hydrated `productsfeed` ("Turn-Around" group, each option carrying
  colorspec/runsize + price). The route now fetches base prices + feed in
  parallel and joins them into a valid colorspec × runsize × turnaround
  matrix with markup.
- Live quote (`/api/4over/quote`) now accepts repeated `?options=<uuid>` so
  add-on options (grommets, etc.) change the price. Verified: grommets
  "Corners Only" $4.02 → $4.58.

## 3. Markups
- Per-category markup multipliers live in `lib/4over-config.ts`
  (1.35×–1.6×). Business Cards 1.5×, specialty/Majestic 1.55–1.6×, most
  others 1.4×. Retail = wholesale × multiplier. (Wholesale matches 4over
  exactly since it's the same API the reference uses.)

## 4. Supabase database + admin
- Created a new Supabase project; ran migrations via the SQL Editor using a
  combined file `scripts/_combined_migrations.sql` (001–006 + 008).
  **007 is intentionally excluded** — it conflicts with 006
  (`fourover_categories` etc. using `fourover_uuid` vs the code's
  `category_uuid`/`product_uuid`), which caused
  "column fourover_uuid does not exist".
- Created a test admin headlessly with the service-role key:
  `admin@web2print.com` / `Admin12345!` (`is_admin = true`). Login verified.

## 5. Vercel build fixes
- `lib/stripe.ts`: wrapped the Stripe client in a Proxy so it's created on
  first use, not at module load (was crashing the build with
  "Neither apiKey nor config.authenticator provided" when env vars are absent
  at build time).
- `app/api/4over/setup-tables`: moved the Supabase client into the handlers
  (`force-dynamic`) for the same reason.
- General rule adopted: never instantiate clients that read env at module
  scope in route files — do it lazily inside handlers.
- `app/(storefront)/print/[category]/page.tsx`: category landing pages no
  longer call `createClient()` up-front, so `/print/<parent>` renders even
  when DB env vars aren't set.

## 7. Product configurator (`components/print/product-configurator-client.tsx`)
- Renders **all** of a product's option groups dynamically (Size, Stock,
  Coating, Sides, Quantity, Turnaround + extras like Orientation, Shape,
  Grommets, H-Stakes, Flute, Lamination, Foil Color, …) as dropdowns.
- Live pricing via the quote endpoint including the selected extra
  option_uuids; hidden/duplicate groups keep their default selection so the
  price stays correct.
- De-duplicate option groups by name (the feed can repeat a group like
  "Foil Color") — fixes duplicate rows + React key errors.
- `allowedGroups` prop: **Signs & Banners** is limited to Size, Shape, Stock,
  Sides, Grommets, Quantity, Turnaround Time. Business Cards and others show
  their full set.
- **Add to Cart**: writes the selection (product/colorspec/runsize/turnaround
  uuids + option_uuids + labels + price) to `localStorage["print_cart"]` in
  the shape the `/cart` page reads.

## 11. Sandbox ordering — proven (not yet wired into checkout)
- Full order submission to the 4over sandbox succeeded via `/api/4over/orders`:
  order_status "Success", payment authorized (dummy profile token
  `1010101010`), status "Awaiting Artwork".
- Key payload facts: `job_name` is a **top-level** job field (not nested in

  the order pending artwork; `shipper` needs `shipping_method` + the shipping
  quote's `service_code`; `is_test_order: true` for sandbox.
